MG-PickerStudio 使用推荐
发表于2020-07-14|更新于2020-07-26|神秘研究
|字数总计:2.3k|阅读时长:7分钟|阅读量:

前言

  前几个月，我通过一些比较简单的方式破解了 MGTools 和 Animbot 两个动画插件。
  最近有动画师想要用 MG-PickerStudio 这款很不错的插件， B 站上有插件的官方教程 链接

  这个插件所有功能都可以免费试用，但是不付费的话无法保存当前编辑好的界面。
  这就是缺点。
  我之前想到自己已经破解了 MGTools ，相信这个破解也不会那么困难。

  然而事情并没有我想象的那么简单，MG-PickerStudio 的代码升级了。
  反而比起 MGTools 更难破解。

  MGPicker 的 mll 复杂了很多，很多窗口界面直接用 C++ 的 Qt 写的，因此也没法用上次提到的临时目录锁定删除的方式获取到隐藏的 Mel 界面。
  Animbot 的解锁方式也无法对抗无法保存的硬逻辑。
  最后经过我不少时间的折腾，只能使用最底层的方法，通过反汇编的方式修改二进制实现破解。

  对于反汇编，其实我并不是一无所知，但是也研究不深入。
  早在高中的时候，有个叫 GameAssassin 的软件可以截取游戏运行中的骨骼模型资源。
  但是开发公司已经倒闭了，想要付费也无法使用，那个时候，我就和哥哥一起研究过反汇编，想要试图破解这个软件。
  最后当然是以失败告终了，毕竟完全没人带，那个时候又不会汇编，只是拿着百度贴图的一些 OD 的教程学了带你皮毛。
  不过倒是有所收获的，让我认识了一些破解的基础手法。

破解尝试

  开始经过 VScode 的全盘搜索，发现注册界面的代码完全找不到。
  然后通过 procmon 查询文件的写入写出，也没有发现有任何有价值的 注册表 信息写入写出，以及文件的读写。
  所以最后只能从 mll 入手了，毕竟 mll 其实就是 dll ，只不过 Maya 特殊配置入口的 dll 而已。
  由于我还没有学过 Qt dll 插件的开发，尚不清楚 Qt 的 dll 和 Maya 的 mll 有什么关系。
  极大可能并没有什么关系，毕竟 Maya 早期的版本并不使用 Qt

  破解 dll 大部分网上的推荐 IDA 和 ollydbg ， 一个静态分析，一个动态分析。
  动态分析要比 IDA 的静态分析强大很多，毕竟可以追踪程序的流转，可以快速定位需要破解的入口。
  但是 mll 并不是标准的 dll 文件，因此最后我使用 IDA 进行静态分析破解。

IDA 破解尝试

  一开始完全不熟悉 IDA 的使用，踩了不少的坑。
  最后我总结一套比较行之有效的方法，这其中也参考了网上一些视频的经验。

  IDA 可以静态分析出 mll 代码块，并且可以转换为图表的形式，查看程序运行的逻辑可谓相当直观。

  汇编语言对我来说实在是太难了，有很多微小的操作，我还得一行一行研究 CPU 内部运行的逻辑去推算变量就非常麻烦。
  看了晚上一个教程 链接 ，感觉非常受用。
  先通过程序内部封装的字符串开始寻找索引。
  按 Shift+F12 可以罗列出 mll 内部封装的字符串。
  从中可以发现 MGPicker 很多隐藏 UI 的命名都在其中。

  找到对应的字符串上按 x 键可以罗列出这个 字符串 被引用的地方。
  然后就可以一路进行程序溯源。自己也是个新手，所以这个溯源过程踩了很多坑。

  破解的思路有两个，一个是暴力破解，毕竟点击保存会弹出注册窗口，那我通过一些手段绕开这个注册窗口代码，直接执行保存的代码即可。
  第二个思路就比较高级，那就是要编写注册机实现万能破解。

  我最后实现的只是 暴力破解 ，毕竟还是菜鸟，但是对于注册机的逆向工程还是挺感兴趣，有机会的话一定要深入学习一下，以后程序相关的东西就可以白嫖了。

IDA 暴力破解

  关于暴力破解，其实有两种方案可以尝试。
  一种就是分析 Register 点击按钮的时候对哪些底层变量进行了操作，然后让程序的默认变量设置到那个值。
  但是这个方法真的太难了，我溯源的时候由于不太能看懂汇编语言，只能重点留意含有字符串的代码。
  这样我比较好理解这一大段的代码大致是在干什么的，好在界面的处理还是比较好追溯的。

  但是上面的方案，我已经追溯到 注册成功 和 注册失败 的 代码块了。
  我通过 Patch 工具，强行将注册失败的引导线引导 注册成功 的地方。
  在 Maya 里面已经成功点击 Register 弹出注册成功的窗口，并且关闭注册界面，但是其实工具还是处于没有注册的状态。
  点击保存依然会弹出注册窗口。

  于是我只好寻找第二个方案，触发保存的时候，到底是哪一断汇编导致注册弹出的出现。
  这个由于没有经验，也没有懂行的人来带我，这个我只能用时间堆砌去寻找蛛丝马迹。
  功夫不负有心人，最后通过我找到一个 sub 地址在好多个地方都被调用的，然后这个赋值 al 寄存器上。
  然后这个后面会进行 test al,al 来判断寄存器的值是否为零进行跳转。

  通过我的修补工具将 jz 改为 jmp 强制跳转到 al 为 1 的状态。
  后面就完美实现了 mll 程序的破解了。

总结

  溯源的过程其实非常麻烦，基本上就是不断按 X 键溯源，然后错了就按 Esc 键返回。
  由于基本知识关注字符串信息，所以这个不起眼的 test al,al 并没有怎么引起我的关注。
  直到后来，我多次发现在不同的地方触发报错都看到了 test al,al 之后才逐渐怀疑这个 al 寄存器的值是否是目标。

  后来网上发现是有个 sub 函数进行 al 寄存器值的判断，于是尝试修改才最终误打误撞实现了破解，这个研究花费了我好几天。
  为此还看了小甲鱼的汇编教程，学习了一些比较基础的汇编命令，还是看得很头疼，毕竟自己没有动手去验证。

  不顾如果想要学好 C++ ，估计 C 和 汇编 是逃不了的。

  最后成功破解 2017 的 mll 之后，我想再接再厉继续用差不多的方法去破解 2019 mll
  结果并不成功， 2019 mll 的 VS 编译升级了，IDA 溯源更加困难，没法单纯通过字符串进行追踪到那一段判断的代码了。
  不过好在 MG-Picker 这个保存功能只要破解一个版本就 OK 了，没有其他版本顶多就是麻烦一点。

  如果我逆向再深入一点，开发一个 Keygen 才是一劳永逸的操作。
  最后我在网上找到了 CrackMe 合集，来自吾爱破解的。
  以后想要深入学习可以深入研究一下下面的两个网址。

  • crackme 160
  • github crackme 160

补充 2020-7-15

  今天顺手吧 2018 和 2019 也破解了。
  虽然IDA的追踪索引不好用，但是通过 2017 的破解，我知道这个核心判断是和 al 寄存器相关的。
  可惜 IDA 无法实现多行搜索。
  如果直接搜索 test al,al 这个汇编代码，范围就太多了。
  于是我将IDA生成的汇编全部拷贝到 VScode 里面。
  然后将通过 VSCode 进行多行匹配，查找 test al,al 下一行接 retn 的。
  得到的结果就只有10多个。
  后面简单排查一下，就找到了核心的判断位置，顺利把 2018 和 2019 也破解了。
